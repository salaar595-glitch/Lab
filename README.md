Сценарій A - SQL Injection
Як відтворити (небезпечна версія)

Bash
curl "http://localhost:3000/api/v1/posts/search-unsafe?title=' OR '1'='1"
Причина - рядок вставляється напряму в SQL без екранування, що дозволяє обійти фільтрацію і повернути всі пости з бази даних.
Як виправлено - всі запити в файлі post.repository.js тепер використовують виключно параметризований синтаксис зі знаком "?" (prepared statements).
Як перевірити що більше не відтворюється

Bash
curl "http://localhost:3000/api/v1/posts/search?title=' OR '1'='1" -H "X-Demo-UserId: 1"
Очікуваний результат - система повертає порожній масив {"items":[]}, тобто ін'єкція більше не спрацьовує і сприймається як звичайний текст.

Сценарій Б - IDOR (Insecure Direct Object Reference)
Як відтворити (небезпечна версія)

Bash
curl -X DELETE http://localhost:3000/api/v1/posts/1 -H "X-Demo-UserId: 2"
Раніше сервер повертав статус 204 No Content і успішно видаляв пост, дозволяючи користувачу під id 2 видалити об'єкт, який належить користувачу під id 1, через повну відсутність перевірки прав доступу на рівні об'єкта.
Як виправлено - додано обов'язкове поле ownerUserId у таблиці posts і comments. Створено проміжне програмне забезпечення checkOwner.middleware.js, яке перевіряє відповідність id поточного сеансу із id власника ресурсу перед виконанням операцій PUT або DELETE. При виявленні невідповідності сервер миттєво блокує запит.
Як перевірити що більше не відтворюється

Bash
curl -X POST http://localhost:3000/api/v1/posts -H "Content-Type: application/json" -H "X-Demo-UserId: 1" -d '{"title":"Мій пост","category":"Інше","body":"Тест 123","author":"User1"}'
curl -X DELETE http://localhost:3000/api/v1/posts/1 -H "X-Demo-UserId: 2"
Очікуваний результат - при спробі видалення чужого поста користувачем 2 сервер повертає статус 403 Forbidden.

Сценарій В - XSS (Cross-Site Scripting)
Як відтворити (небезпечна версія)
Раніше функція render() використовувала властивість innerHTML для динамічної вставки отриманих даних у такий спосіб: tr.innerHTML = <td>${post.title}</td>. Якщо в змінну post.title передавався шкідливий рядок на кшталт зміненого тегу зображення з помилкою, це призводило до негайного виконання стороннього JavaScript коду в браузері жертви.
Як виправлено - повністю замінено логіку у функціях render() та loadComments() на безпечні методи document.createElement та textContent. Будь-які отримані від користувача дані тепер інтерпретуються системою суто як текст і ніколи не вставляються в DOM дерево як виконуваний HTML код.
Як перевірити - створити новий пост, де замість назви title вказати скрипт у тегу зображення. Очікуваний результат - назва безпечно відображається на сторінці у вигляді звичайного тексту, а вбудоване вікно alert не спрацьовує.

Auth (401/403)
Повна відсутність обов'язкового заголовка X-Demo-UserId у запиті викликає помилку 401. Передача невідомого або неіснуючого у базі даних значення userId так само викликає помилку 401. Будь-яка спроба модифікації або видалення чужого ресурсу авторизованим користувачем викликає помилку 403.
